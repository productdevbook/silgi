## `$route({ path })` now works with the client

Previously, procedures using `$route({ method: 'GET', path: '/api/orders' })` were only reachable via the custom REST path. The typed client (`createClient`) always called the tree path (`POST /commerce/orders/list`), which returned 404. [#1](https://github.com/productdevbook/silgi/issues/1)

Now the client resolves custom routes at call time using `extractRoutes()`:

```ts
// server/routes.ts — lightweight, no server code
import { extractRoutes } from 'silgi'
import { appRouter } from './router'
export const routes = extractRoutes(appRouter)
```

```ts
// client.ts
import { createLink } from 'silgi/client/ofetch'
import { routes } from './server/routes'

const link = createLink({
  url: 'http://localhost:3000',
  routes,
})
```

`extractRoutes()` walks the router tree and produces a minimal nested object containing only `{ path, method }` per procedure — no resolve functions, guards, or schemas leak into the client bundle.

### Path parameter substitution

`:param` placeholders in custom routes are automatically extracted from the input and placed in the URL:

```ts
// Server
const byId = s.$route({ method: 'GET', path: '/api/users/:id' }).$resolve(({ params }) => db.users.find(params.id))

// Client
const user = await client.users.byId({ id: '42' })
// → GET /api/users/42
```

Used params are removed from the request body — only remaining fields are sent.

See [Client docs — Route resolution](/docs/client#route-resolution) for the full guide.

## `createSafeClient()`

A new proxy wrapper that makes every procedure return `{ error, data }` instead of throwing:

```ts
import { createSafeClient } from 'silgi/client'

const client = createSafeClient<AppRouter>(link)
const { error, data, isError, isSuccess } = await client.users.list()
```

No more wrapping individual calls in `safe()` — the entire client is safe by default. See [Client docs — Error handling](/docs/client#error-handling).

## Client-side OpenTelemetry

New `withOtel()` link plugin creates spans for every client call:

```ts
import { withOtel } from 'silgi/client/plugins'
import { trace } from '@opentelemetry/api'

const link = withOtel(baseLink, {
  tracer: trace.getTracer('my-frontend'),
})
```

Each span is named `rpc.client/{procedure}` (e.g. `rpc.client/users.list`) with `rpc.system: silgi` attribute. Errors record an `exception` event automatically. Uses the same `Tracer`/`Span` interface as the server-side `otelWrap`. See [OTel plugin docs](/docs/plugins/otel#client-side-tracing).

## Stream consumption utilities

New `silgi/client/consume` export with lifecycle-driven stream consumption:

```ts
import { consumeIterator, consumeWithReconnect, mapIterator } from 'silgi/client/consume'

// Basic consumption with callbacks
await consumeIterator(iterator, {
  onEvent: (data) => updateUI(data),
  onError: (err) => reportError(err),
  onFinish: () => cleanup(),
  signal: controller.signal,
})

// Auto-reconnection with lastEventId resumption
await consumeWithReconnect({
  connect: (lastEventId) => client.events.subscribe(undefined, { lastEventId }),
  onEvent: (data) => updateUI(data),
  maxReconnects: 10,
  reconnectDelay: 2000,
})
```

See [Client docs — Consuming streams](/docs/client#consuming-streams).

## WebSocket client link

New `WSLink` for type-safe bidirectional RPC over WebSocket:

```ts
import { WSLink } from 'silgi/client/ws'

const link = new WSLink({ url: 'ws://localhost:3000/ws' })
const client = createClient<AppRouter>(link)
```

Handles connection lifecycle automatically — connects on first call, multiplexes concurrent calls over a single socket. Call `link.dispose()` to close. See [WebSocket docs — Client](/docs/protocols/websocket#client).

## More client improvements

- **Async `DynamicLink`**: Selector can now return `Promise<ClientLink>` — useful for lazy-loading links via dynamic `import()`
- **Retry-After header**: `withRetry` now respects `Retry-After` headers from 429/503 responses by default. Supports both delay-seconds and HTTP-date formats. Disable with `respectRetryAfter: false`
- **Streaming batch option**: `BatchLink` accepts a `streaming` option for progressive response delivery (server-side NDJSON support coming in a future release)
