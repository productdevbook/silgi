[Nitro](https://nitro.build) is the server engine behind Nuxt and TanStack Start. Silgi integrates with Nitro using the `serverEntry` config — Silgi becomes the server framework.

## Setup

Create a `server.ts` that exports a `fetch` handler:

```ts twoslash title="server.ts"
// @noErrors
import { silgi } from 'silgi'
import { z } from 'zod'

const s = silgi({
  context: (req) => ({
    db: getDB(),
    headers: Object.fromEntries(req.headers),
  }),
})

const health = s.$resolve(() => ({ status: 'ok' }))

const listUsers = s
  .$input(z.object({ limit: z.number().optional() }))
  .$resolve(({ input, ctx }) => ctx.db.users.findMany({ take: input.limit }))

const appRouter = s.router({ health, users: { list: listUsers } })

export default { fetch: s.handler(appRouter) }
```

Point Nitro to it:

```ts title="nitro.config.ts"
import { defineNitroConfig } from 'nitro/config'

export default defineNitroConfig({
  compatibilityDate: '2025-01-01',
  serverEntry: './server.ts',
})
```

That's it. No file-system routing, no adapters — Silgi IS the framework.

```
POST /health          → { status: "ok" }
POST /users/list      → [{ id: 1, name: "Alice" }, ...]
```

<Callout>Nitro detects the `fetch` method on the default export and routes all requests through it.</Callout>

## Nuxt

For Nuxt apps, use the same `server.ts` entry. Add it to your Nuxt config:

```ts title="nuxt.config.ts"
export default defineNuxtConfig({
  compatibilityDate: '2025-01-01',
  nitro: {
    serverEntry: './server/server.ts',
  },
})
```

## Coexisting with existing routes

When you add Silgi to an app that already has its own routes — GraphQL endpoints, REST API routes, file-system routes — the default `fetch` export takes over **all** requests. Silgi returns a `404` response for any procedure it doesn't recognize, which prevents Nitro/Nuxt from handling those routes normally.

To fix this, wrap the handler and return `undefined` for unmatched routes. This tells Nitro to continue with its normal routing pipeline:

```ts twoslash title="server.ts"
// @noErrors
import { silgi } from 'silgi'

const s = silgi()

const appRouter = s.router({
  health: s.$resolve(() => ({ status: 'ok' })),
})

const silgiHandler = s.handler(appRouter)

export default {
  fetch: async (request: Request) => {
    const response = await silgiHandler(request)
    // Let Nitro handle routes that silgi doesn't know about
    if (response.status === 404) return
    return response
  },
}
```

With this pattern, requests matching silgi procedures go through silgi. Everything else — your GraphQL endpoint, Nitro API routes, Nuxt pages — continues to work as before.

<Callout>Only use this pattern when you need to mix silgi with other routes. For apps where silgi handles everything, the simpler `export default { fetch: s.handler(appRouter) }` is fine.</Callout>
