## Procedures

Every API endpoint is a **procedure**. Three kinds:

| Kind | HTTP Method | Use Case |
|---|---|---|
| `query` | GET | Read data, cacheable |
| `mutation` | POST | Write data, side effects |
| `subscription` | SSE/WS | Real-time streaming |

<Tabs items={['Short form', 'Short + input', 'Full config']}>
  <Tab value="Short form">
  ```ts
  const health = query(() => ({ status: "ok" }))
  ```
  </Tab>
  <Tab value="Short + input">
  ```ts

    z.object({ id: z.number() }),
    ({ input }) => db.users.get(input.id), // [!code highlight]
  )
  ```
  </Tab>
  <Tab value="Full config">
  ```ts
  const createUser = mutation({
    use: [auth, timing],           // middleware
    input: z.object({ name: z.string() }),
    output: UserSchema,            // output validation
    errors: { CONFLICT: 409 },     // typed errors
    resolve: ({ input, ctx, fail }) => { // [!code highlight]
      if (exists(input.name)) fail("CONFLICT")
      return db.users.create(input)
    },
  })
  ```
  </Tab>
</Tabs>

## Guards

Guards run **before** the handler. Their return value merges into `ctx`.

```ts
const auth = guard(async (ctx) => {
  const user = await verifyToken(ctx.headers.authorization)
  if (!user) throw new KatmanError("UNAUTHORIZED")
  return { user, permissions: user.permissions } // → ctx.user, ctx.permissions
})
```

<Callout type="info" title="Why guards are fast">
Guards are **flat functions** — no `next()` callback, no closure per request. This is why katman's pipeline is 5.7x faster than oRPC's onion middleware.
</Callout>

Stack multiple guards — context accumulates:

```ts
const proc = mutation({
  use: [auth, rateLimit, adminOnly],
  resolve: ({ ctx }) => {
    ctx.user        // from auth guard
    ctx.rateLimit   // from rateLimit guard
    // adminOnly throws if not admin — no return needed
  },
})
```

## Wraps

Wraps are **onion-style** — they run code before and after the handler:

```ts
const timing = wrap(async (ctx, next) => {
  const start = performance.now()
  const result = await next() // [!code highlight]
  console.log(`${performance.now() - start}ms`)
  return result
})
```

### Execution order

```
Request
  → Guard 1 (auth)          merges { user }
  → Guard 2 (rateLimit)     merges { rateLimit }
  → Wrap 1 (timing)         before ↓
    → Wrap 2 (errorBoundary) before ↓
      → resolve()            handler runs
    → Wrap 2                  after ↑
  → Wrap 1                    after ↑
Response
```

## Lifecycle Hooks

Global hooks — powered by [hookable](https://github.com/unjs/hookable):

```ts
const k = katman({
  context: (req) => ({}),
  hooks: {
    request: ({ path, input }) => console.log(`→ ${path}`),
    response: ({ path, durationMs }) => console.log(`← ${path} ${durationMs.toFixed(1)}ms`),
    error: ({ path, error }) => console.error(`✗ ${path}`, error),
    "serve:start": ({ url }) => console.log(`Server at ${url}`),
  },
})
```

Dynamic registration:

```ts
const unhook = k.hook("request", ({ path }) => metrics.inc(path))
// later...
unhook() // remove the hook
```

<Callout>
Hooks have a **sync fast-path** — when all hooks are synchronous, zero Promises are allocated. No performance cost when hooks aren't registered.
</Callout>

## Type Inference

Types flow end-to-end — no codegen needed:

```ts twoslash
import { katman } from "katman"
import { z } from "zod"

const k = katman({ context: () => ({}) })

const appRouter = k.router({
  health: k.query(() => ({ status: "ok" as const })),
  users: {
    list: k.query(
      z.object({ limit: z.number().optional() }),
      ({ input }) => [{ id: 1, name: "Alice" }].slice(0, input.limit),
    ),
  },
})

type AppRouter = typeof appRouter
//   ^?
```
