Three packages, each with a single responsibility:

| Package        | Responsibility           | Dependencies                   |
| -------------- | ------------------------ | ------------------------------ |
| `api-contract` | Shared types and schemas | `silgi`, `zod`                 |
| `api-server`   | Implements the contract  | `api-contract`, `silgi`, `zod` |
| `frontend`     | Consumes the API         | `api-contract`, `silgi`        |

## 1. Define the contract

The contract package exports API shape definitions — input/output schemas, route metadata, and error codes. No server logic.

```ts title="packages/api-contract/src/index.ts"
import { contract } from 'silgi/contract'
import { z } from 'zod'

export const UserSchema = z.object({
  id: z.string(),
  name: z.string(),
  email: z.string().email(),
})

export const api = contract({
  users: {
    list: {
      type: 'query',
      input: z.object({ limit: z.number().min(1).max(100).optional() }),
      output: z.array(UserSchema),
      route: { method: 'GET', path: '/api/users' },
    },
    create: {
      type: 'mutation',
      input: z.object({ name: z.string().min(1), email: z.string().email() }),
      output: UserSchema,
      errors: { CONFLICT: 409 },
      route: { method: 'POST', path: '/api/users' },
    },
  },
})

export type AppContract = typeof api
```

<Callout>
The contract uses `contract()` from `silgi/contract` — a lightweight function that returns the definition as-is, providing type inference without any runtime behavior.
</Callout>

## 2. Implement on the server

The server package imports the contract and provides real implementations. Types are enforced at compile time.

```ts title="packages/api-server/src/server.ts"
import { silgi } from 'silgi'
import { implement } from 'silgi/contract'
import { api } from 'api-contract'

const s = silgi({
  context: (req) => ({
    db: getDB(),
    headers: Object.fromEntries(req.headers),
  }),
})

// implement() ensures your handlers match the contract's types
const router = implement(api, {
  users: {
    list: async ({ input, ctx }) => {
      return ctx.db.users.findMany({ take: input.limit ?? 10 })
    },
    create: async ({ input, ctx, fail }) => {
      if (await ctx.db.users.exists(input.email)) fail('CONFLICT')
      return ctx.db.users.create(input)
    },
  },
})

s.serve(router, { port: 3000 })
```

## 3. Generate route metadata

For REST-style routing, generate a `routes.json` at build time:

```ts title="packages/api-server/scripts/extract-routes.ts"
import { minifyContractRouter } from 'silgi/contract'
import { api } from 'api-contract'
import { writeFileSync } from 'fs'

const routes = minifyContractRouter(api)
writeFileSync('../frontend/src/routes.json', JSON.stringify(routes, null, 2))
console.log('✓ Routes extracted')
```

This outputs only `{ path, method }` metadata — zero business logic.

## 4. Consume on the frontend

The frontend imports the contract type and the routes JSON. No server code in the bundle.

```ts title="packages/frontend/src/client.ts"
import { createClient } from 'silgi/client'
import { createLink } from 'silgi/client/ofetch'
import type { AppContract } from 'api-contract'
import routes from './routes.json'

const link = createLink({
  url: 'http://localhost:3000',
  routes, // Enables REST path resolution (GET /api/users, POST /api/users)
})

const client = createClient<AppContract>(link)

// Full autocomplete and type checking
const users = await client.users.list({ limit: 10 })
```

<Callout type='info'>
Use `import type` for the contract. This ensures the import is erased at build time — no server code is bundled into the frontend.
</Callout>
