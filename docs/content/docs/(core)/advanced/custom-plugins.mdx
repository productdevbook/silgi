Silgi has no special plugin API. A "plugin" is a guard, a wrap, a set of lifecycle hooks, or a combination — packaged for reuse. This page shows how to build and share your own.

## Custom guard — tenant isolation

A guard that reads a tenant ID from the request and adds it to the context. Every downstream procedure and guard can access `ctx.tenant`.

```ts twoslash
// @noErrors
import { silgi, SilgiError } from 'silgi'

const s = silgi({
  context: (req) => ({
    headers: Object.fromEntries(req.headers),
  }),
})

// The guard
export const tenantGuard = s.guard(async (ctx) => {
  const tenantId = ctx.headers['x-tenant-id']
  if (!tenantId) {
    throw new SilgiError('BAD_REQUEST', { message: 'Missing X-Tenant-Id header' })
  }

  const tenant = await db.tenants.findById(tenantId)
  if (!tenant) {
    throw new SilgiError('NOT_FOUND', { message: 'Tenant not found' })
  }

  return { tenant } // ctx.tenant is now typed
})
```

Use it in any procedure:

```ts twoslash
// @noErrors
const listUsers = s.$use(tenantGuard).$resolve(({ ctx }) => {
  // ctx.tenant is typed — TypeScript knows it exists
  return ctx.db.users.findMany({
    where: { tenantId: ctx.tenant.id },
  })
})
```

### Making it configurable

Wrap the guard in a factory function to accept options:

```ts twoslash
// @noErrors
export function createTenantGuard(options: { headerName?: string; required?: boolean }) {
  const header = options.headerName ?? 'x-tenant-id'

  return s.guard(async (ctx) => {
    const tenantId = ctx.headers[header]

    if (!tenantId && options.required !== false) {
      throw new SilgiError('BAD_REQUEST', {
        message: `Missing ${header} header`,
      })
    }

    if (!tenantId) return { tenant: null }

    const tenant = await db.tenants.findById(tenantId)
    return { tenant }
  })
}
```

## Custom wrap — response caching

A wrap that caches procedure responses in memory. It intercepts the pipeline, checks the cache, and either returns the cached value or calls `next()`.

```ts twoslash
// @noErrors
export function createCacheWrap(
  options: {
    ttl?: number // milliseconds
    keyFn?: (ctx: any, input: any) => string
  } = {},
) {
  const ttl = options.ttl ?? 60_000 // 1 minute default
  const cache = new Map<string, { value: unknown; expires: number }>()

  return s.wrap(async (ctx, next) => {
    const key = options.keyFn ? options.keyFn(ctx, ctx.__input) : JSON.stringify(ctx.__input)

    const cached = cache.get(key)
    if (cached && cached.expires > Date.now()) {
      return cached.value
    }

    const result = await next()

    cache.set(key, {
      value: result,
      expires: Date.now() + ttl,
    })

    return result
  })
}
```

Use it:

```ts twoslash
// @noErrors
const cacheOneMinute = createCacheWrap({ ttl: 60_000 })

const listProducts = s.$use(cacheOneMinute).$resolve(({ ctx }) => ctx.db.products.findMany())
```

<Callout>
  This is an in-memory cache for demonstration. For production, use Redis or a dedicated caching layer. The pattern is
  the same — the wrap checks the cache before calling `next()`.
</Callout>

## Custom lifecycle hooks

For cross-cutting concerns that apply to every procedure, lifecycle hooks are cleaner than adding a wrap to every `.$use()` call.

```ts twoslash
// @noErrors
import { lifecycleWrap } from 'silgi'

export function createMetricsPlugin(metrics: MetricsClient) {
  return lifecycleWrap({
    onStart: ({ ctx }) => {
      metrics.increment('rpc.requests')
    },
    onSuccess: ({ durationMs }) => {
      metrics.histogram('rpc.duration', durationMs)
    },
    onError: ({ error }) => {
      metrics.increment('rpc.errors')
    },
  })
}
```

Apply it per-procedure:

```ts twoslash
// @noErrors
const metrics = createMetricsPlugin(statsd)

const listUsers = s.$use(metrics).$resolve(({ ctx }) => ctx.db.users.findMany())
```

Or apply it globally via instance-level hooks:

```ts twoslash
// @noErrors
const s = silgi({
  context: (req) => ({ db: getDB() }),
  hooks: {
    request: () => metrics.increment('rpc.requests'),
    response: ({ durationMs }) => metrics.histogram('rpc.duration', durationMs),
    error: () => metrics.increment('rpc.errors'),
  },
})
```

## Composing multiple pieces

A real plugin often combines a guard, a wrap, and hooks. Package them together:

```ts twoslash
// @noErrors
export function createAuditPlugin(options: { logger: Logger }) {
  const auditGuard = s.guard((ctx) => {
    return {
      auditTrail: [] as string[],
    }
  })

  const auditWrap = s.wrap(async (ctx, next) => {
    const result = await next()

    if (ctx.auditTrail.length > 0) {
      options.logger.info({
        action: ctx.__procedurePath,
        user: ctx.user?.id,
        trail: ctx.auditTrail,
      })
    }

    return result
  })

  return { auditGuard, auditWrap }
}
```

Use both:

```ts twoslash
// @noErrors
const { auditGuard, auditWrap } = createAuditPlugin({ logger })

const deleteUser = s.$use(auth, auditGuard, auditWrap).$resolve(({ input, ctx }) => {
  ctx.auditTrail.push(`Deleted user ${input.id}`)
  return ctx.db.users.delete(input.id)
})
```

## Packaging for distribution

If you want to publish your plugin as an npm package:

1. Export factory functions, not instances — let consumers pass their own `silgi` instance or options
2. Use TypeScript generics so the guard/wrap types compose with any context
3. Document what the plugin adds to `ctx` and what it expects

```ts twoslash
// @noErrors
// silgi-plugin-tenant/index.ts
import type { SilgiInstance } from 'silgi'

export function createTenantPlugin<T extends SilgiInstance>(s: T) {
  const tenantGuard = s.guard(async (ctx) => {
    // ...
    return { tenant }
  })

  return { tenantGuard }
}
```

Consumers install and use it:

```ts twoslash
// @noErrors
import { createTenantPlugin } from 'silgi-plugin-tenant'

const { tenantGuard } = createTenantPlugin(s)
```
