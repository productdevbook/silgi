Every database query is a potential bottleneck. The Drizzle integration wraps your Drizzle instance so every `select`, `insert`, `update`, and `delete` appears as a span in silgi analytics ŌĆö with duration, SQL text, table name, and operation type. No manual instrumentation needed.

## Setup

Install both packages:

<Tabs items={['pnpm', 'npm', 'bun']}>
  <Tab value='pnpm'>```bash pnpm add silgi drizzle-orm ```</Tab>
  <Tab value='npm'>```bash npm install silgi drizzle-orm ```</Tab>
  <Tab value='bun'>```bash bun add silgi drizzle-orm ```</Tab>
</Tabs>

Wrap your Drizzle instance with `instrumentDrizzle`, then use `withCtx` inside procedures to connect queries to the current request trace:

```ts twoslash title="server.ts"
// @noErrors
import { silgi } from 'silgi'
import { instrumentDrizzle, withCtx } from 'silgi/drizzle'
import { drizzle } from 'drizzle-orm/node-postgres'
import * as schema from './schema'

const db = instrumentDrizzle(drizzle(process.env.DATABASE_URL!, { schema }), { dbName: 'myapp' })

const s = silgi()

const listUsers = s.$resolve(async ({ ctx }) => {
  return withCtx(ctx, () => db.select().from(schema.users))
})

export default { fetch: s.handler(s.router({ users: { list: listUsers } })) }
```

## What gets captured

Each query produces a span with these attributes:

| Attribute        | Type      | Description                                            |
| ---------------- | --------- | ------------------------------------------------------ |
| `db.system`      | `string`  | Database system identifier (default: `postgresql`)     |
| `db.name`        | `string`  | Logical database name (e.g. `auth`, `ecommerce`)       |
| `db.operation`   | `string`  | SQL operation (`SELECT`, `INSERT`, `UPDATE`, `DELETE`) |
| `db.statement`   | `string`  | Full SQL text (truncated to `maxQueryTextLength`)      |
| `db.transaction` | `boolean` | `true` when the query runs inside a transaction        |
| `net.peer.name`  | `string`  | Database host                                          |
| `net.peer.port`  | `number`  | Database port                                          |

## Span naming

Spans are named using the pattern `db.{operation}.{table}`:

```
db.select.user          ŌĆö SELECT from user table
db.insert.order         ŌĆö INSERT into order table
db.update.product       ŌĆö UPDATE on product table
db.delete.session       ŌĆö DELETE from session table
db.tx.select.user       ŌĆö SELECT inside a transaction
db.tx.insert.order_item ŌĆö INSERT inside a transaction
```

This is more useful than generic names like `db.query` ŌĆö you can immediately see which table and operation is slow in your analytics dashboard.

## Configuration

All options are optional. Pass them as the second argument to `instrumentDrizzle`:

| Name                 | Type      | Default        | Description                                           |
| -------------------- | --------- | -------------- | ----------------------------------------------------- |
| `dbName`             | `string`  | `undefined`    | Logical database name. Appears as `db.name` attribute |
| `dbSystem`           | `string`  | `'postgresql'` | Database system identifier                            |
| `captureQueryText`   | `boolean` | `true`         | Include SQL text in spans                             |
| `maxQueryTextLength` | `number`  | `1000`         | Truncate SQL text after this many characters          |
| `peerName`           | `string`  | `undefined`    | Database host. Appears as `net.peer.name`             |
| `peerPort`           | `number`  | `undefined`    | Database port. Appears as `net.peer.port`             |

## Multiple databases

Instrument each database separately with a distinct `dbName`:

```ts twoslash title="db.ts"
// @noErrors
import { instrumentDrizzle } from 'silgi/drizzle'
import { drizzle } from 'drizzle-orm/node-postgres'

const authDb = instrumentDrizzle(drizzle(process.env.AUTH_DATABASE_URL!), {
  dbName: 'auth',
  peerName: 'db.example.com',
  peerPort: 5432,
})

const ecommerceDb = instrumentDrizzle(drizzle(process.env.ECOMMERCE_DATABASE_URL!), {
  dbName: 'ecommerce',
  peerName: 'db.example.com',
  peerPort: 5432,
})
```

In your analytics, spans from each database are distinguished by the `db.name` attribute.

## Transactions

Transaction queries are automatically detected. They get a `db.tx.` prefix in the span name and the `db.transaction: true` attribute:

```ts twoslash title="server.ts"
// @noErrors
import { withCtx } from 'silgi/drizzle'

const createOrder = s.$resolve(async ({ ctx, input }) => {
  return withCtx(ctx, () =>
    db.transaction(async (tx) => {
      // span: db.tx.insert.order
      const [order] = await tx.insert(schema.orders).values(input).returning()
      // span: db.tx.insert.order_item
      await tx.insert(schema.orderItems).values(input.items.map((item) => ({ ...item, orderId: order.id })))
      return order
    }),
  )
})
```

## Security

By default, the full SQL text is captured in spans. If your queries contain sensitive data (PII, tokens, secrets), disable SQL capture in production:

```ts twoslash
// @noErrors
import { instrumentDrizzle } from 'silgi/drizzle'
import { drizzle } from 'drizzle-orm/node-postgres'

const db = instrumentDrizzle(drizzle(process.env.DATABASE_URL!), {
  dbName: 'myapp',
  captureQueryText: process.env.NODE_ENV !== 'production',
})
```

You still get span names, durations, and operation types ŌĆö just not the raw SQL.

<Callout type='warn'>
  `withCtx` is required. Without it, queries execute normally but are invisible to silgi analytics. Always wrap your database calls inside `withCtx(ctx, () => ...)` in your procedures.
</Callout>
